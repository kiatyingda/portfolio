01 — The Problem

A marketplace problem
disguised as a UX problem.

For Passengers

Hard to get a ride. Post-COVID supply crunch meant too many passengers, too few drivers. Users were constantly cancelling and rebooking different services trying to get a car.

For Grab

Most users only booked JustGrab (the flagship service). Other services had idle drivers. Fulfillment was dropping.

02 — What the Data Said

The demand to switch
was already there.

78%
of users who couldn't get a ride rebooked within 30 minutes.
90%
among those, booked a service at a similar price point.

Users were already willing to switch — they were just doing it manually, with a lot of friction. The opportunity was to remove that friction.

03 — My Approach

Three lenses.
One direction.

Existing Data

The rebook behavior stats (78% / 90%) confirmed willingness to switch services. The signal was already in the product.

User Research

Partnered with a researcher and content designer to test 6 different concepts across Singapore and Indonesia.

Competitive Analysis

DiDi lets users multi-select services, but adds cognitive load — more options = more overwhelm. A useful warning sign.

04 — Strategy

Two horizons.
One framing.

Short-term

User mental-model shift

Change the way users book — from single-selection to multiple-selection. Remove the friction of manual rebooking by making multi-select the default affordance.

Long-term

Intent-based experience

Built on tradeoffs between price/time and user needs. Users choose by category, not by service name. The system matches them to the best available option.

I framed this as a two-horizon strategy to get alignment from product leadership.

05 — Concepts Explored

6 concepts. Each testing
a different hypothesis.

Manual selection
User wants maximum control and comparison
Fare slider
User only cares about fare, not what's selected
Bundles
User doesn't want to spend time choosing
Post-booking prompt
User won't multi-select until they've tried and failed
Intent-based (needs)
User's mental model is needs-first (e.g. "I have kids")
Intent-based (categories)
Simplifying options reduces drop-off

06 — What Users Said

I was wrong
about one thing.

01

"I'm more used to seeing the list of services over just categories."

Users prefer familiar patterns over novel abstractions. Don't innovate the mental model when the task is already high-stakes.

02

"I will only book my regular service, and won't add more unless I really can't get a car."

Multi-select is a fallback behavior, not a default one. Design for the context in which the need arises.

03

"I don't like horizontal scrolling. I want to scroll and compare prices of all the services."

Vertical list with price comparison is essential. Scanning and comparing is the core behavior.

The intent-based categories concept I was most excited about didn't resonate. Users wanted control, not abstraction.

07 — The Constraint

The decision that shaped
the entire trajectory.

The Problem

Engineering needed at least 6 months to rebuild the allocation engine for true multi-selection. We couldn't wait that long without validating demand.

My Move

I worked closely with product and engineering to implement the feature in phases — rolling out low-cost MVPs to validate product-market fit and get learnings while the backend caught up.

08 — Execution

Three phases.
Ship → break → fix.

Phase 1 · 2021 · Hacky Experiment

"Any Nearby Ride"

A single new service that matched users to the first available driver across multiple ride types. No allocation engine change needed.

User books → system searches across ride types → first available driver is matched → user is told which ride type they got via chat message.

The UI wasn't polished (pre design system), but the concept worked.

✓ Good adoption rate and increased fulfillment rates. Validated product-market fit.

Phase 2 · Post-Booking Prompt

When drivers were scarce, prompt users
to expand their search.

A bottomsheet prompted users to add more ride types to increase their odds after booking.

What went wrong: Real users complained they got allocated a service they didn't book — a trust violation. The feature sometimes didn't appear. 0.8% of users froze when the prompt appeared.

What I fixed: Replaced opaque bundles with explicit ride type names. Rewrote the copy: "Few drivers now. Add other ride types to your search?" Added a timer-based auto-proceed CTA.

✓ Reduced freezing behavior. Improved trust through explicit labeling.

Phase 3 · The Full Solution

Pre-booking multi-select —
true user control.

Checkboxes on the ride type list. Users explicitly choose which services they'll accept before booking. Combined with the refined post-booking prompt as a safety net.

Visual refinements: smaller service icons for tighter density, checkbox moved to the right for better scannability.

✓ This was the version that required the rebuilt allocation engine. True user control + system flexibility.

10 — Navigating Stakeholders

Same constraint.
Less friction.

GrabShare couldn't support multi-selection. The PM wanted a disruptive modal dialog. I pushed back.

Option A Shipped

Contextual disable

Show GrabShare in the list but prevent it from being checked alongside other services — disable checkboxes contextually. No modal, no interruption to booking momentum.

Option B Rejected

Modal dialog

"GrabShare cannot be selected with other ride types. Proceed to unselect?" An extra step that breaks the booking flow at a critical moment.

11 — The Vision

Where I was
pushing the product.

An intent-based booking experience where services are grouped by user need — Standard, Economy, Fastest — rather than internal product names. Users choose a category, and the system matches them to the best available option.

I'd been advocating for this since early research. The phased approach was always designed to move users toward this model over time.

12 — Results + Reflections

What moved.
What I learned.

+0.4pp
Fulfilment rate — matched rides across Southeast Asia
+$1M
Incremental revenue per year
1

Marketplace design insight

The biggest UX lever was often a backend architecture decision. My job was to make the phased approach feel intentional to users, even when it was a constraint.

2

Copy matters as much as UI

The content rewrite in Phase 2 drove as much improvement as the interface changes. I should have invested in copy testing earlier instead of treating it as a polish step.

3

Users accept less control if you give more transparency

When we showed explicit ride types instead of opaque bundles, trust went up even though we were asking for the same thing.
